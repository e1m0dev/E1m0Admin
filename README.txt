🧑‍💻 E1m0dev: На деле - Систем куча, плагин старается и выдерживать архитектуру цепей по типу:
❓ | Архитектура ЦЕПЕЙ:
    Для обычных команд: PLAYER -> COMMAND (Адаптер) -> SERVICE (Принятие решений, обработка абстракций) -> REPOSITORY (Принятие факта, ведение диалога с базой данных) -> DAO (Структурирование и подготовка | Доступ) -> DATABASE (Хранение информации | MAIN PLUGIN STATE, там - только факты системы);
    Для например репорта: PLAYER -> GUI -> CONTROLLER -> SERVICE -> REPOSITORY -> DAO -> DATABASE -> EVENT -> LISTENER -> ACTIONS
    А если это emergency? - PLAYER -> COMMAND -> COMMAND-CONTROLLER -> REPOSITORYDAO -> DATABASE -> EVENT -> LISTENER -> ACTIONS
    В пин-коде - тоже самое: PLAYER -> GUI -> CONTROLLER -> SERVICE -> REPOSITORY -> DAO -> DATABASE -> EVENT -> LISTENER -> ACTIONS


‼️| Архитектура*:
    Мои:
        Главные управляющие абстракции: API;
        Главные точки входа: COMMANDS;
        Обработчики логики: SERVICE;
        Ведущие диалог с базой: REPOSITORY;
        Адаптирующиеся под условия: DAO;
        Хранящие значения: DATABASE;
        Вызывающиеся: Events;
        Слушающие: LISTENERS;
        Хранящие: STATE;
        Подсказывающие: COMPLETERS;
        Помогающие: UTILS;
        Регулируемые: CONFIGS;

API:
     Есть 4 слоя абстракции плагина:
         GUI;
         REPO;
         SERVICE;
         UTILS;

     GUI:
        ReportGuiAPI: Позволяет взаимодействовать с репортами из вне, то есть из другого плагина например.
            Пример: Сейчас там метод: openReportGUI | Позволяет открывать меню сразу с сообщением из другого плагина.

        SecretCodeGuiAPI: Позволяет взаимодействовать с репортами из вне, то есть из другого плагина например.
            Пример: Сейчас там метод: openReportGUI | Позволяет открывать меню сразу с сообщением из другого плагина.

     REPO:
        GUI:
            ReportSystemRepositoryAPI: Отвечает за репорты а точнее - внутри системы.
                Пример: В нем есть метод на данный момент как: gameReportSend, именно он распределяет обязанности транспортировки моего DAO запроса по цепи после контроллера и сервиса, то есть он - исходит от запроса системы по прихоти клиента - по этому и System.

            SecretCodeRepositoryAPI: Отвечает за обработку пин-кодов для администрации в GUI, то есть он тоже системный и исходит от контроллера. Почему не SecretCodeSystemRepositoryAPI? Название получилось бы слишком длинным и выходило за рамки приличия, по этому я решил оставить как есть.
                Пример: В нем есть методы управления секретными кодами на момент создания такие как: staffSetSecretCode, systemDeleteAdmin, systemSetSecretCode, getSecretCode, все они - расписаны внутри.

        PLUGIN:
            GameRepositoryAPI: Отвечает за все что связанно в игре с администратором и базой данных его игровой в общем то процесс, репорты != игровая логика, это уже отдельная система.
                Пример: Если когда нибудь понадобится логировать rewatch logs администратора -> это пойдет туда. Сейчас я считаю такое небольшим перегрузом, по этому он сейчас пустует, НО, существует для будущих обновлений.

            StaffRepositoryAPI: Абсолютно все что связанно с администратором и его старшим.
                Пример: Повышения, понижения, удаления, постановление, бонусы и все, все, все, но тут передача с базой через DAO

            SystemRepositoryAPI: Системы которые связаны с системой администрации.
                Пример: Посмотреть зарплату игрока, его префикс, его уровень админки или состоит ли он вообще в администрации?

    Service:
        GUI:
            ReportSystemServiceAPI: Отвечает за взаимодейсвтие между системами
                Пример: В нем есть метод на данный момент как: clickToReport. Он передает данные из GUI контроллера в сервис для обработки бизнес логики самого Backend.

            SecretCodeServiceAPI: Отвечает за клики и логику, за нее отвечает не сколь контроллер сколь сам сервис, потому что именно он хранитель бизнес логики, а контроллер - удобный адаптер
                Пример: В нем есть методы "Дистанционного доступа" к меню SecretCode: oneStepHandler, который можно вызвать игроку сразу же с первой цифрой кода которую можно узнать вторым методом getInputCode, продолжая дальше идти по цепи логики.

        PLUGIN:
            ConsoleServiceAPI: Отвечает за все что связанно в игре с администратором.
                Пример: Все взаимодействия а точнее обработка логики связанные с консолью.

            GameServiceAPI: Отвечает за все что связанно в игре с администратором связанный с игровыми взаимодействиями включая обработку логики, его игровой в общем то процесс, репорты != игровая логика, это уже отдельная система но она имеет место в самом администраторе, то есть это его и игра и задача, по этому репорты обычно в двух местах в отдельных ReportService и в GameService
                Пример: Реватч, невидимка, репорты, и прочие функции.

            StaffServiceAPI: Абсолютно все что связанно с администратором и его старшим.
                Пример: Повышения, понижения, удаления, постановление, бонусы и все, все, все.

            SystemServiceAPI: Системы контроллеров систем
                Пример: handleReportAccept, это системный контроллер точки входа в команду: ReportAcceptController.

    UTILS:
        PLUGIN:
            PermissionsManagerAPI: Отвечает естевственно за права администратора в игре.
                Пример: checkSecretCodeAccess буквально вопрос: Есть ли у администратора доступ к админке?

            GameServiceAPI: Отвечает за все что связанно в игре с администратором связанный с игровыми взаимодействиями включая обработку логики, его игровой в общем то процесс, репорты != игровая логика, это уже отдельная система но она имеет место в самом администраторе, то есть это его и игра и задача, по этому репорты обычно в двух местах в отдельных ReportService и в GameService
                Пример: Повышения, понижения, удаления, постановление, бонусы и все, все, все. Но тут передача логики и ее обработки через внутренности

COMMANDS:
    Каждые из команд разделены так же на свои отделы ответственности:
        Admin -> Команды для администрации.
        Staff -> Команды для обычных игроков.
        Player -> Команды для обычных игроков.
        System -> Системные команды больше контроллеры и отправные точки СИСТЕМЫ
        Console -> Контрольные точки которые можно задавать из вне не заходя в игру по средствам самих точек или что самое интересное: API - А это уже сайты, веб, приложения и прочее.

    Admin:
        AccessCommand: Первая точка входа. Без нее админ - просто игрок, без ввода секретного пароля он ничем не отличается от обычного игрока и все права у него заблокированы.
        ReportCommand: Команда-GUI. Открывает свод репортов которые появились за последнее время.
        RewatchCommand: Точка слежки. Команда позволяет следить за игроками от 3-его лица.
        InvisibilityCommand: Команда для использования спец невидимости, прямо "Из коробки", никаких партиклов и эффектов, бесконечная невидимка и все тут..

    Staff:
        AdminBonusAllCommand: Выдать бонус ВСЕМ администраторам который указан внутри.
        AdminSetSecretCode: Установить пин-код администратору если его еще нет в базе.
        AdminDeleteCommand: Уволить администратора от лица стафф сотрудника сервера.
        AdminBonusCommand: Выдать бонус ОДНОМУ администратору который указан внутри.
        AdminDownCommand: Понизить администратора от лица стафф сотрудника сервера.
        AdminSetCommand: Поставить администратора от лица стафф сотрудника сервера.
        AdminUpCommand: Повысить в доходяги администратора от лица стафф сотрудника.

    Player:
        PlayerReportCommand: Позволяет отправить новый репорт.

    System:
        ReportAcceptController: Обычный контроллер arepaccept, который перехватывает репорт и начинает работать через сервис, то есть обычная прокладка которая слушает как Listener.

    Console:
        ConsoleSetSecretCommand: Установить пин-код администратору от лица API если его еще нет в базе.
        ConsoleDownAdminCommand: Понизить администратора от лица консоли сервера.
        ConsoleDelAdminCommand:  Уволить администратора от лица консоли сервера.
        ConsoleSetAdminCommand:  Поставить администратора от лица консоли сервера.
        ConsoleUpAdminCommand:   Повысить администратора от лица консоли сервера.

SERVICE:
    Сервисы главные обработчики логики, не важно это бизнес логика или внутриигровая, именно на них сгружена ответственность думать и принимать решения.
        Есть несколько основных сервисов:

    GUI:
        ReportSystemService: Некоторые действия которые связаны с репортами, только некоторые! Которые не касаются гейма но касаются системы по этому ReportSYSTEMService
        SecretCodeService: Исполнение действий из меню сразу в "Just In Time" почти никогда не нужно oneStepHandler, но он заполняет цепь, хотя скорее всего я его уберу.

    PLUGIN:
        AdminsStaffService: Выполнение все что связано  с:  Стафф -> Админ.     Связан с: StaffServiceAPI
        AdminSystemService: Выполнение все что связано  с:  Система -> Система  Связан с: SystemServiceAPI
        AdminGameService: Выполнение все что связано    с:  Админ -> Игра.      Связан с: GameServiceAPI
        ConsoleService: Выполнение все что связано      с:  Консоль -> Админ.   Связан с: ConsoleServiceAPI

REPOSITORY:
    Если сервисы у меня принимают решения - репозитории ведут диалог с базой и записывают через DAO сразу в базу окончательные заключения логики если значение прошло все проверки Service.
        Есть несколько основных репо как и сервисов:

    GUI:
        ReportSystemRepository: gameReportSend -> Как главный из методов позволяет отправить факт о закрытом репорте, соеденен с ReportDAO.
        SecretCodeRepositoryAPI: SET, SETSYSTEM, GET, RESET, все как у людей, отправят как захочешь, легко подстраиваются связаны с SecretCodeDAO

    PLUGIN:
        AdminSystemRepository: Логирование фактов по обычным системным транзакциям по типу adminPay или handleReportAccept.
        AdminStaffRepository: Логирование фактов по взаимодействию с составом, set, del, up и так далее по накладной.
        AdminGameRepository: Факты системы для логирования ДЕЙСТВИЙ администратора в ИГРЕ а не в системе.

DAO:
    Мои обработчики обьектов в базе данных, очень способные и полезные при работе цепь POINT-SERVICE-REPO-DAO-DATA.

    PLUGIN:
        SecretCodeDAO: Работа с пин кодами администрации;
        AdminsDAO: Работа с администраторами в Базе Данных;
        ReportDAO: Работа с репортами в Базе Данных;
        BonusDAO: Работа с логированием бонусов а Базе Данных;

DATABASE:
    Без базы данных очень сложно держать структуру а с ней в плагине я использовал и кэширование и трансфер в базу.

    PLUGIN:
        DatabaseSource: Установщик связи с драйверами, HickariCP, HickariDataSource и прочим через конфиг и JDBC ну или коннектор базы данных.
        DatabaseManager: На основе DatabaseSource, создает и управляет таблицами в установленной связью базе данных.

EVENTS:
    Личные и приватные ивенты плагина которые очень помогают в разгрузке сервисов и упрощении кода в сервисах перекладывая ответственность на распределяющую Event прокладку Listener.

    E1m0:
        AdminAccessEvent: Когда администратор регистрируется в системе -> Вызвается ивент.

LISTENERS:
    Подразделены на несколько слоев ответственности чтобы не путать холодное с теплым:

    Bukkit:
        JoinListener: Когда игрок заходит на сервер, там сразу проверяется статус администратора, его сессии, права, т.д, можно сделать систему синхронизации API для защиты.
        QuitListener: Когда игрок выходит из сервера, там если он админ - удаляется его State, AdminSession, и отбираются права администратора.

    E1m0:
        AdminAccessListener: Когда администратор подтверждает себя как полноправного владельца прав, срабатывает триггер.

STATE:
    Мои не наглядные состояния, есть так же несколько типов хэширования данных в системе:
        Admin;
        Report;
        Session;
        SecretCode;

    Admin:
        Admin: Record с snapshot immutable состоянием администратора который позже mutable менеджером перетекает.
            AdminManager: Помогает справлятся с состоянием Admin через mutable рычаги виксируя diff изменения, прямо как в React

    Session:
       AdminSession: А именно сюда и перетекает STATE состояние через mutable manager;
           AdminSessionManager: А этим констракт методом управляется состояние, для JoinListener && QuitListener у AdminSessionManager есть методы: join и quit, которые заполняют и очищают кэш память админ сессий.

    Report:
        Report: Report а вот это уже - не постоянное состояние, и оно не может быть immutable, хоть он и имеет mutable состояние он все равно живет от силы 20 секунд и умирает передаваясь уже в DAO и в Database так что состояние не долговечное а значит надежное

    SecretCode:
        SecretCodeState: Тоже вроде понятно, тот же mutable state class который управляет цепью порядка SecretCode, да возможно не так "профессионально" через сеттеры состояний mutable Manager в Immutable состоянии State но система все равно надежна.
            SecretCodeManager: Управление состоянием разрешений, выдать когда нужно, забрать когда не нужно, проверить когда спросят - красота.
